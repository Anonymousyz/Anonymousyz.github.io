# 个人主页方法体系重构设计

- 日期：2026-07-20
- 状态：设计已确认，待实施计划
- 选择：A（身份位于首屏之后）+ A2（中文为主、英文旁注）+ M1（方法双栏索引）

## 1. 背景

当前首页以“研究与判断、写作、学习、产品与工程”四项并列展示方法。四项并非同一层级：研究和产品工程是两类工作，写作是表达与交付方式，学习贯穿所有工作。读者能看到四项能力，却看不到它们如何共同处理一个问题。

Notion 中已经形成更完整的结构：研究与证据支持判断；多元认知、思维模型和人物视角用于检查判断；写作负责把结论交给具体读者；产品、视觉和工程负责把业务要求做成可使用的系统；学习贯穿以上过程。

本次重构要把这套关系呈现出来，同时保持首页简洁。项目案例暂不扩写。

## 2. 目标与读者

主要读者是招聘经理、业务负责人和技术负责人。他们应在较短时间内看清三件事：

1. 作者长期处理什么问题；
2. 作者怎样从研究走到判断，又怎样从业务走到系统；
3. 哪些方法有项目、工具或公开材料支持。

首页继续保留两句主标题：

> 从分析到决策
> 从技术到应用

网站不设置明显的 FDE 求职页，不使用“能力矩阵”“个人 OS”“认知引擎”“赋能”等包装性名称。

## 3. 首页结构

首页按以下顺序组织：

1. 导航；
2. 首屏；
3. 身份；
4. 方法；
5. 项目；
6. 公开作品；
7. 页脚。

### 3.1 首屏

- 眉题改为“个人方法与项目实践”，删除含义不清的“产业研究、产品与软件”。
- 主标题保持“从分析到决策 / 从技术到应用”。
- 简介只写真实经历和网站收录内容，不加入求职口号。

建议简介：

> 过去八年，我做过产业研究、政策分析和重大项目，也参与产品与软件系统建设。这里收录我使用的方法、公开作品和项目记录。

### 3.2 身份

身份区位于首屏之后，使用两栏排版。首页不再为身份补充自我证明，只展示身份、引文和出处。

#### 长期主义者

中文主引文：

> 无欲速，无见小利。欲速则不达，见小利则大事不成。
> ——孔子《论语·子路》

英文旁注：

> It's All About the Long Term.
> ——Jeff Bezos，1997 Letter to Shareholders

英文原句是 1997 年亚马逊股东信中的章节标题。出处：<https://ir.aboutamazon.com/files/doc_financials/annual/2002_shareholderLetter.pdf>

#### 终身学习者

中文主引文：

> 学不可以已。
> ——荀子《劝学》

英文旁注：

> Stay hungry. Stay foolish.

页面署名应写为“Whole Earth Catalog，Steve Jobs 2005 年斯坦福演讲引述”，避免把原句错误归为 Jobs 原创。出处：<https://news.stanford.edu/stories/2005/06/steve-jobs-2005-graduates-stay-hungry-stay-foolish>

中文引文承担主要视觉重量。英文使用更小字号和较低对比度，作为补充线索。

### 3.3 方法

方法区采用 M1 双栏索引，不绘制流程图、闭环图或飞轮。

顶部先放一个通栏入口：

- **持续学习**：遇到不熟悉的问题，先找可靠资料，再放进实际工作里检验。

下方分为两条同等重要的主线。

#### 从分析到决策

1. **研究与证据**：先弄清谁要做什么决定，再确定查什么、查到哪一步。
2. **多元思维**：换一种解释、一个模型或一个专业视角，检查原来的判断。
3. **写作与表达**：把结论、依据和需要谁做什么写清楚。

#### 从技术到应用

1. **产品定义**：从使用者、场景和流程开始定义产品。
2. **视觉与信息设计**：用层级、排版和状态帮助人更快看懂、做对。
3. **工程与交付**：把规则做进数据、权限和测试，直到系统可以交付。

七个入口全部可点击。首页只保留标题和一句说明，详细内容进入子页面。

### 3.4 项目与公开作品

- 项目继续使用“一个长案例、两个 Demo”的结构。
- 本轮不扩写工业绿色微电网评价软件、总体所技改平台和口腔小程序案例。
- 三个公开 GitHub 工具继续作为可检查的公开作品。
- 不新增 LinkedIn 入口。

## 4. 方法总页与路由

保留四个已有地址，防止旧链接失效；新增三个页面：

| 展示名称 | 路由 | 处理方式 |
|---|---|---|
| 持续学习 | `/methods/learning/` | 保留路由，重写内容 |
| 研究与证据 | `/methods/research-and-judgment/` | 保留路由，调整标题与内容 |
| 多元思维 | `/methods/plural-thinking/` | 新增 |
| 写作与表达 | `/methods/writing/` | 保留路由，调整标题与内容 |
| 产品定义 | `/methods/product-definition/` | 新增 |
| 视觉与信息设计 | `/methods/visual-information-design/` | 新增 |
| 工程与交付 | `/methods/product-and-engineering/` | 保留路由，调整标题与内容 |

预计静态路由由 11 个增加到 15 个：新增方法总页和三个方法子页。

### 4.1 方法总页结构

方法总页使用独立 H1“方法”，不重复首页首屏。页面依次包含：

1. 一段简短说明，交代这套方法来自研究、项目和公开工具；
2. “持续学习”通栏入口；
3. “从分析到决策 / 从技术到应用”双栏索引；
4. 每条主线各选一个项目和一个公开作品作为延伸阅读；
5. 一段内容边界，说明页面记录当前做法，后续会随实际工作修改。

首页方法区负责快速浏览，方法总页负责说明各部分之间的关系。两处使用同一份方法数据，避免名称、顺序和链接漂移。

## 5. 子页面内容边界

每个方法页面使用相同的阅读顺序，但不套用同一组机械标题。页面需要回答：

- 这类工作通常从什么问题开始；
- 作者实际怎样处理；
- 工作会留下什么材料、决定或系统；
- 方法在哪个项目或公开作品中使用；
- 哪些做法后来发生过变化。

### 5.1 持续学习

内容包括：

- 从真实任务确定学习范围；
- 优先查权威资料和原始来源；
- 通过写作、讨论和动手实现检查理解；
- 保留术语、来源、易错点和已验证例子；
- 把上一个项目留下的知识带到下一个项目。

页面不使用“进入陌生领域”“认知升级”“学习闭环”等表达。

### 5.2 研究与证据

内容包括：

- 从决策问题确定研究范围；
- 区分事实、解释、评价和建议；
- 核对时间、定义、口径和来源立场；
- 记录适用范围、未知项和需要补充的证据；
- 说明工业绿色微电网项目中标准如何进入指标和计算规则。

### 5.3 多元思维

内容包括：

- 从问题出发选择模型，不罗列模型名称；
- 比较另一种同样说得通的解释；
- 根据风险调用学科、人物和专业视角；
- 对重要、难以撤回的决定保留推理记录；
- 使用反方问题检查假设、因果和盲区。

页面可以使用“专家委员会”一词，但要明确其含义：这里指人物视角与提问机制。涉及专业结论时，由相应专业人员复核。页面不暗示真实专家为作者或项目背书。

### 5.4 写作与表达

内容包括：

- 区分探索性写作和结论性写作；
- 先确认读者、用途和待决定事项；
- 分开事实、解释、评价和建议；
- 写清行动主体、条件和边界；
- 根据误读情况先修改标题和结构。

### 5.5 产品定义

内容包括：

- 识别使用者、场景、问题和现有替代方案；
- 画清角色、对象、流程和异常路径；
- 用真实行为和付出判断问题强度；
- 选择一个关键假设，裁剪 MVP；
- 定义验证指标、产品范围和停止条件。

### 5.6 视觉与信息设计

内容包括：

- 把视觉作为降低理解成本的工具；
- 从页面任务确定焦点、层级和阅读顺序；
- 使用颜色、字体、间距和组件规则保持一致；
- 覆盖 Word、PPT、数据表格、网页和软件界面；
- 检查对比度、状态完整性、移动端和可访问性；
- 使用真实改稿或界面问题说明方法怎样变化。

页面不把视觉能力写成审美宣言，也不把工具使用等同于设计能力。

### 5.7 工程与交付

内容包括：

- 将业务规则落实到数据、状态、权限和校验；
- 根据项目约束选择架构和技术方案；
- 覆盖核心流程、异常输入、权限和恢复路径；
- 使用测试、构建检查和发布门禁控制质量；
- 写清监控、回滚、数据和维护边界；
- 区分内部功能版本、正式 UAT、预生产和生产验收。

## 6. 视觉规则

- 延续现有暖纸、宋体、大标题、深绿色外框和细线体系。
- 身份区使用两栏，中文引文为主，英文旁注字号更小。
- 方法区使用通栏“持续学习”加左右两列的方法索引。
- 索引依靠编号、线条和留白建立层级，不改成卡片墙。
- 动效仅限链接和索引行的轻微状态反馈。
- 桌面端身份和方法均使用双栏；移动端改为单栏。
- 移动端顺序：首屏、长期主义者、终身学习者、持续学习、从分析到决策、从技术到应用、项目、公开作品。

## 7. 内容与可信度约束

- 不新增未经材料支持的经历、成果、指标或客户背书。
- 项目状态继续区分内部版本、测试记录、专业复核和正式验收。
- 方法页面要链接到相关项目或公开作品；没有证据的地方不写成已形成的成熟能力。
- 首页不出现“证明能力”“接受公开检查”“形成完整闭环”等自证语句。
- 中文文案使用具体人物、动作和对象，避免抽象名词连续堆叠。

## 8. 实现边界

本轮实施包括：

- 首页身份区和方法区重构；
- 方法总页；
- 三个新方法页面；
- 四个既有方法页面重写；
- 导航、SEO、sitemap、测试和构建断言同步更新；
- 已有 LinkedIn 删除改动纳入最终验证。

本轮不包括：

- 项目案例深度扩写；
- 新增项目截图或附件；
- 改写 Git 历史；
- 推送 GitHub 或发布网站；
- 改造三个公开工具仓库。

## 9. 验证标准

实施完成后至少执行：

1. 项目现有测试；
2. Astro 静态构建；
3. 15 个预期路由的构建产物检查；
4. sitemap 检查；
5. 内部链接、锚点和外部仓库链接检查；
6. LinkedIn 文本与结构化数据扫描；
7. 1440px、768px 和 390px 视口检查；
8. 键盘导航、可见焦点、文本对比度和横向溢出检查；
9. 浏览器控制台错误检查；
10. 对首页和七个方法入口做一次中文措辞、逻辑和 AI 腔复审。

完成上述检查只说明本地版本具备提交条件，不代表已经推送或发布。

## 10. 已确认事项

- 公开定位采用“个人方法体系”，但首页不使用“个人 OS”等自我包装。
- 两条主线保持同等重要。
- 身份采用“长期主义者 / 终身学习者”。
- 身份位于首屏之后。
- 中文经典为主，英文名言作为旁注。
- 学习入口名称为“持续学习”。
- “多视角审查”改为“多元思维”。
- 方法采用双栏索引。
- 案例内容留到后续单独完善。
