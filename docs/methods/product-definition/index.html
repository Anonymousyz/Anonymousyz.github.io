产品做什么 · 为什么做 · 验证什么

做产品时，我先回答三件事：做什么、为什么做、验证什么。先判断问题是否值得解决，再还原使用者的真实任务，用范围可控的方案检验关键假设。确认后，再把对象、流程、权限和验收条件交给工程设计。

四个阶段各自回答什么

每个阶段都要回答不同问题，也有自己的退出条件。上一阶段留下的未知项会进入下一阶段，但不能用原型完成度代替问题证据，也不能用工程投入倒逼产品判断。

经营判断
说明为什么现在值得投入、服务谁，以及不处理会付出什么代价。
问题验证
确认真实任务、现有做法、发生条件和问题影响。
方案验证
用范围可控的试验检查价值、可用性和关键风险。
工程化设计入口
把已确认的对象、流程、规则、异常和验收条件交给工程。

先做经营判断

经营判断先写清目标使用者、问题发生的场景、当前代价、可选替代方案和投入理由。团队还要说明成功以后改变什么，以及什么结果会让投入停止。

机会来源可以是政策、技术、组织变化或一线需求。任何来源都要落到具体问题和承担者，趋势名称本身不能支持立项。

还原真实任务

从最近一次真实经历开始：事情怎样触发，使用者拿到什么信息，经过哪些步骤，在哪里等待、返工、求助或放弃。访谈中的偏好需要与行为记录互相核对。

同时记录现有替代方案，包括表格、聊天、电话、纸面登记和人工协调。新产品需要说明它减少了什么成本，又增加了什么学习与切换成本。

把角色、对象和流程写清

先识别谁发起、谁办理、谁审核、谁查看结果；再识别他们共同操作的业务对象、字段和状态。角色与对象不清，页面容易按部门名称堆成菜单。

主流程之外，还要记录资料缺失、审核退回、重复提交、转交失败和权限不足等异常。异常如何被发现、谁能处理、从哪里恢复，属于产品定义的一部分。

验证问题是否成立

问题验证要回答发生条件、影响程度、当前代价和解决意愿。观察、访谈、历史记录和小样本数据可以互相校对，避免用一次强烈意见概括全部场景。

证据不足时，把它写成待验证假设，并说明下一步观察什么。问题、对象和价值已经达到当前决策所需的清晰度以后，再进入方案比较。

用小方案验证关键假设

MVP 用较低成本检查最危险的假设。可以先用原型验证理解和流程，也可以保留人工环节，先确认数据与决策是否有用。

试验前写清覆盖对象、观察指标和停止条件。结果只支持实际覆盖的场景，一次演示不能直接支持全面上线。

把验证结果交给工程

进入工程化设计前，交接内容包括确认过的问题、目标使用者、核心对象、流程与异常、范围表、验收条件、未决事项和不做清单。技术团队据此讨论数据、权限、接口和运行约束。

验证结果偏离预期时，先修改问题定义、范围或方案。增加功能需要新的理由与验证，不能用开发量掩盖前一阶段的失败。

使用说明

适用边界

  • 产品方法可以减少错误投入，市场和组织是否接受仍需实际验证。
  • 访谈、观察和指标各有偏差，需要结合具体场景解释。
  • MVP 的结果只适用于实际覆盖的对象和条件，扩大范围前应重新验证。
