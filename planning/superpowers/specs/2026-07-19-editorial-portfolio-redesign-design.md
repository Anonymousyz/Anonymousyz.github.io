# 编辑式个人主页改版设计说明

日期：2026-07-19
状态：已实现；主案例长文与两个 Demo 的最终素材仍待补充
范围：主页、全站视觉、四个方法页；项目长案例与 Demo 正文另行打磨

## 1. 目标

网站面向招聘面试官、潜在合作方和从 GitHub 进入的读者。访客应在短时间内看懂两条并重的主线：

> 从分析到决策<br>
> 从技术到应用

页面通过方法、公开作品和真实项目呈现个人特点，不出现“FDE 求职页”“能力证明”“适配岗位”等求职话术，也不把研究、写作、学习和工程拆成互不相干的能力标签。

改版后的首页只回答三个问题：

1. 这个人主要做什么；
2. 他平时怎样工作；
3. 有哪些内容可以继续阅读。

思想和方法放进子页，项目证据放进案例。首页不承担完整论证。

## 2. 不做什么

- 不制作显式的 FDE 求职落地页；
- 不使用“四项能力”“工作闭环”“能力系统”等抽象框架；
- 不使用证据徽章、成熟度标签、深色 SaaS 面板或仪表盘式布局；
- 不在首页反复解释公开边界和严谨性；
- 不公开客户材料、真实业务数据、未授权截图、账号、路径、凭据和私有代码；
- 本轮不完成工业绿色微电网长案例和两个 Demo 的最终正文、截图与媒体编排。

## 4. 首页内容

### 4.1 首屏

眉题：

> 产业研究、产品与软件

主标题保持用户确认的原文，两行字号、字重和颜色一致：

> 从分析到决策<br>
> 从技术到应用

引言：

> 过去八年，我做过产业研究、政策分析和重大项目，也参与产品与软件系统建设。这里收录我常用的方法、公开工具和三个项目的工作记录。

首屏不放按钮组、交付状态和侧边信息板。

### 4.2 方法

栏目说明：

> 这些页面记录我怎样研究、写作、学习和做软件。

四个入口使用不同句式，不为了整齐写成同一模板：

1. **研究与判断**
   遇到一个研究题目，我先问清楚谁要据此做决定，再确定查什么、查到哪一步。
2. **写作**
   动笔前，我会先确认写给谁、希望对方看完做什么，再安排内容和结构。
3. **学习**
   碰到不熟悉的领域，我先从手头的问题做起。做不下去、讲不清楚，就回头补知识。
4. **产品与工程**
   做系统之前，我会先弄清谁在用、工作怎么流转、哪些规则不能错，再定页面、接口和技术方案。

### 4.3 项目

栏目说明：

> 一个完整案例，两个脱敏 Demo。

主案例：**工业绿色微电网评价软件**

> 这套软件把政策和评价标准拆成数据要求、计算规则和报告内容。我参与前期研究、产品设计和交付检查；案例也会写明哪些部分已经测试，哪些还需专业复核。

侧栏只保留两项：

- 我的工作：研究、产品设计、测试与交付检查
- 案例内容：从评价标准到软件功能

Demo 01：**总体所技改平台项目**

> 把项目材料、角色和审批流程做成系统中的台账、权限和工作台。

Demo 02：**口腔小程序项目**

> 设计家长授权、检查报告、预约和后台协作流程。

### 4.4 公开作品

栏目说明：

> 我把工作中常用的研究和交付方法整理成了三个公开工具。

- **Research-to-Decision Toolkit**
  从研究问题、证据整理到方案比较的一组模板。
- **AI Prototype-to-Production Toolkit**
  帮助团队检查 AI 原型能否进入真实业务流程。
- **Awesome AI Production Readiness**
  收录 AI 系统上线前需要检查的工程和治理资料。

首页不展示版本、运行环境、命令、证据类型和方法边界。这些内容留在作品页。

### 4.5 关于与页脚

首页删除单独的“关于”大区块，避免重复首屏。页脚保留姓名、GitHub、LinkedIn 和邮件入口；完整经历继续放在 `/about/`。

## 5. 方法页

方法页采用文章结构，不使用统一的“核心命题、方法框架、能力证明、适用边界”模板。每页包含一段真实项目中的使用经历，以及一次后来修改过的做法。

### 5.1 研究与判断

页首：

> 我做研究时，会先问清楚这份材料最后要帮助谁做什么决定。准备立项材料和解释行业趋势，查资料的方法、结论的写法都不一样。

栏目：

- **研究要回答什么**：确认谁要使用、何时使用，以及最后要判断什么；只补充会影响判断的材料。
- **材料怎样处理**：分开记录事实、解释和建议；遇到矛盾时核对时间、定义、口径和立场。
- **结论怎样写**：交代依据、范围和需要重新判断的条件；材料不足时给出下一步补证动作。
- **项目里怎样使用**：以工业绿色微电网评价软件说明政策与标准怎样拆成数据、计算和报告。
- **后来改过哪些做法**：额外寻找一种同样说得通的解释，提前写下可能推翻当前判断的证据，项目结束后核对原有假设。
- **相关工具和案例**：连接 Research-to-Decision Toolkit 与微电网案例。

### 5.2 写作

栏目：

- 写给谁
- 准备解决什么
- 怎样安排结构
- 怎样改稿
- 读者没有看懂时怎么办
- 项目里怎样使用

正文以专报、方案和项目材料的实际写作过程为主。事实、解释、判断和建议可以分开讲，但不把它们包装成四步模型。修改过程应写出读者反馈如何改变标题、结构或行动表述。

### 5.3 学习

栏目：

- 从手头的问题开始
- 先补哪部分
- 怎样检查是否学会
- 怎么带到下一个项目
- 后来改过哪些做法

正文避免“进入陌生领域”“认知更新”“学习迁移”等术语。重点写遇到不会的内容时怎样查资料、做出一个结果、发现错误并返工。判断是否学会的标准包括能否讲清楚，以及能否在项目中正确使用。

### 5.4 产品与工程

栏目：

- 先把业务说清楚
- 谁在用，事情怎么走
- 规则怎样进入系统
- 怎样检查
- 项目做到哪一步
- 相关项目

正文说明需求、角色、流程、数据、权限、测试和运行责任如何在同一个项目中衔接。用总体所技改平台或口腔小程序中的一个具体流程举例。项目状态写明已经完成的部分和仍待验收的部分，不使用成熟度徽章。

## 6. 视觉系统

### 6.1 基调

采用编辑式研究主页：暖纸背景、中文宋体、细分隔线、低饱和砖红点色和大面积留白。页面应接近个人文集与研究档案，避免产品官网和技术仪表盘的气质。

建议色值：

| 用途 | 色值 |
|---|---|
| 外部画布 | `#ecefee` |
| 纸张背景 | `#f4f1e9` |
| 主文字 | `#20211d` |
| 次文字 | `#686860` |
| 分隔线 | `#2c2d27` |
| 点色 | `#8b412d` |
| 外框 | `#42615a` |

### 6.2 字体与层级

- 中文标题和正文：`Noto Serif SC`、`Source Han Serif SC`、`Songti SC`、宋体回退；
- 导航、编号、英文仓库名和少量元数据：系统无衬线字体；
- 两行主标题完全同级，不通过斜体、颜色或字号制造主次；
- 正文阅读宽度控制在 680–720px；
- 首页用索引行，不用卡片网格。

### 6.3 外框与分隔

- 绿色外框 3–4px，桌面圆角 24–28px；
- 首屏结束可用双线，其余栏目只用单线；
- 不使用渐变、玻璃效果、霓虹色和大面积阴影；
- hover 只改变文字颜色或箭头位置，不做浮起卡片。

### 6.4 响应式与动效

- 桌面端约三屏看完方法、项目和公开作品；
- 850px 以下改为单列，保留标题节奏和清楚的点击区域；
- 390px 宽度下主标题不溢出，项目名和英文仓库名允许自然换行；
- 支持 `prefers-reduced-motion`，页面不依赖滚动动画表达信息。

## 7. 内容与公开边界

| 内容 | 可以公开 | 不公开或不声称 |
|---|---|---|
| 工业绿色微电网评价软件 | 脱敏后的研究过程、指标到软件的转换、测试类型、交付检查 | 客户身份、原始材料、真实数据、凭据；不把自动测试写成标准专业复核或正式验收 |
| 总体所技改平台项目 | 材料到角色、对象、流程、权限和工作台的转换；内部功能版本 | 不声称已完成正式 UAT、预生产或生产验收 |
| 口腔小程序项目 | 授权、报告、预约、随访与后台协作流程；受控 Demo | 未成年人和健康原始数据；不声称已公开生产运行 |
| 公开工具 | 仓库、说明、样例、测试、版本与公开链接 | 不以发布代替采用，不声称跨场景有效性 |

边界说明放在对应项目或作品页中。首页只在需要时用一句事实说明，不设置独立“证据与公开边界”栏目。

## 8. 实现边界

本轮实现包括：

- 重做全站颜色、字体、留白、分隔线和响应式规则；
- 重做首页；
- 新增四个方法页及其共享文章布局；
- 更新导航、页脚、404 和元信息；
- 调整集中内容数据与测试；
- 保留 `/projects/`、`/works/`、`/about/` 和现有公开工具案例；
- 保留 `/systems/` 兼容入口。

本轮暂不包括：

- 微电网长案例的最终图文编排；
- 技改平台和口腔小程序的最终 Demo 页面；
- 未经核对的项目截图；
- commit、push、GitHub Pages 发布。

## 9. 组件与内容组织

集中内容文件继续作为公开文案的单一来源，并新增 `methodTopics`、`featuredCase` 和 `demos`。页面组件保持小而明确：

- `SiteHeader`：导航和当前页面状态；
- `SiteFooter`：外部链接与联系入口；
- `MethodIndex`：首页四个可点击入口；
- `ProjectIndex`：一个主案例与两个 Demo；
- `PublicWorkIndex`：三个公开工具的轻量索引；
- `MethodArticle`：方法页的文章外壳，不强制正文使用相同栏目。

删除或停止使用首页现有的 `DeliveryRail`、`WorkLoop`、证据标签和证明型组件。已有项目/作品详情组件可保留，除非新视觉样式需要小范围改动。

## 10. 验证标准

### 内容

- 主标题严格保持“从分析到决策 / 从技术到应用”；
- 四个方法入口均有真实路由；
- 首页不含 FDE 求职话术、私有路径、运行地址和敏感词；
- 首页不出现“能力系统、工作闭环、赋能、迁移、接受公开检查”等包装用语；
- 项目名称、职责和状态与本说明中的公开边界一致。

### 视觉与可用性

- 在 1440px、768px、390px 三种宽度检查布局；
- 键盘可访问所有导航和入口，focus 状态可见；
- 文字与背景保持可读对比度；
- 无横向溢出，无假链接，无仅靠颜色传达的状态；
- 减少动效偏好生效。

### 工程

- `npm test` 通过；
- `npm run build` 通过；
- `npm run verify:build` 通过；
- 构建产物不包含本地路径、开发地址和敏感信息；
- `.superpowers/` 不进入 Git。

## 11. 对标取舍

本次审校参考 Maggie Appleton、Andy Matuschak、Bret Victor、Simon Willison、Julia Evans 和 Robin Sloan 的个人主页。借鉴的是共同做法：主页只给方向和入口，用具体作品建立可信度，把方法放进文章。本站不复制任何一家的视觉外观或文字表达。
